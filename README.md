1.1	ВВЕДЕНИЕ

Объектная модель (ОМ) Sigla Vision / FineBI предназначена для представления объектов системы, их текущего состояния и истории изменений, а также взаимодействия объектов между собой.

Данный код решения по сборке на языке SQL для СУБД PostgreSQL позволяет развернуть на вашей инсталляции ОМ путем создания необходимых для работы таблиц, функций и процедур.

Основной принцип разработки и корректировки существующей системной отчётности состоит в максимальной формализации структуры таблиц самого верхнего уровня на основе таблиц системы и таблиц модели других уровней. 

ОМ не вносит никаких изменений в исходные таблицы FineDB и работает исключительно со своими объектами, которая сама и создает. Таким образом, она практически не влияет на скорость работы ПО Sigla Vision / FineBI. 

Таблицы ОМ могут занимать достаточно много места в БД, так как выполняется процесс денормализации данных с целью более простого использования для задач администрирования. В связи с этим важно отслеживать динамику роста объема данных в БД, который используется ОМ.

Дефолтная схема FineDB в коде "Public". 

1.2	Правила именования объектов 
Для удобства работы администраторов все объекты ОМ в решении начинаются с трехбуквенного префикса, где первая буква «z». Таким образом, при сортировке объектов по имени все таблицы ОМ и системных дашбордов находятся ниже основных таблиц FineDB.

На этапе прототипирования было принято условное разделение таблиц на категории по префиксу названий.
1.	"z00_" – вспомогательные таблицы для системы, не относящиеся к ОМ;
2.	"z01_" – первичные таблицы ОМ (данные полученные из основных таблиц FineDB);
3.	"z10_" – вспомогательные таблицы для ОМ (данные, используемые в расчётах с ручными загрузками: списки, связи, справочники);
4.	"z11_" – таблицы ОМ верхнего уровня (собираемые из таблиц групп "z01_" и "z10_", которые используются для получения наиболее полных представлений о нескольких объектах со свойствами и установленными связями между ними). Например, таблица дашбордов в «Коллекции» объединяет в себе таблицу иерархии коллекции и таблицу шаблонов отчётов. В дальнейшем для части таблиц был выбран ещё один вариант указания на таблицу верхнего уровня через постфикс названия «_all», при этом используется префикс "z01_".

Для системных отчётов и дашбордов был дополнительно создан префикс таблиц "zsd_" для разграничения на схеме таблиц, как сокращение от «system dashboards».

Внутри отдельных функций используются данные версионирования, которые имеют префикс "zst_" и название одноименной версионируемой таблицы FineDB. Версионная таблица содержит все поля исходной таблицы FineDB и три дополнительный поля: тип операции DDL, дата операции и логин УЗ БД выполнившей операцию.  

1.3	ДАННЫЕ, ФУНКЦИИ, ПРЕДСТАВЛЕНИЯ

Код создания (DDL) приведен в файлах sql:
1) для функций в каталоге "zom_functions"
2) для таблиц в каталоге "zom_tables"
3) для вью в каталоге "zom_views"

Наполнение таблиц справочников данными содержатся в каталоге "zom_data" в файлах с префиксом DATA.

Вспомогательные функции, такие как функция преобразования текста в число с обработкой ошибки (try_cast_int), находятся в каталоге "zom_additional".

Основное представление по ОМ это вью v_zom, которое выводит информацию по составу таблиц ОМ и их полям. Другие представления помогают связать вместе таблицы и проверить наличие для них одноименных функций обновления данных. 

Наполнение данными приведено в схожих по имени с названием таблиц функциях. Например, наполнение таблицы z01_ds_all происходит в функции zfr_ds_all.

Разные функции требуют разных комбинаций входных данных:
1) Таблицы FineDB
2) Таблицы LogDB
3) Таблицы справочники ОМ (ручное формирование)
4) Таблицы, формируемые ОМ (производные от вышеперечисленных)
5) Таблицы хранения версионированных изменений FineDB (для отдельных решений, данный раздел будет добавлен после публикации статьи о версионировании данных)

При объединении одинаковых полей из разных таблиц в одну таблицу, в отдельных случаях, был применен подход изменения нейминга полей таблиц источников данных через префикс названия поля вида «xxz_», где «xx» – это сокращение названия исходной таблицы, а «z» – это указание на происхождение поля: либо «m» – сокращение от «main» – для обозначения полей данных, которые являются уже существующими в таблицах БД FineDB, либо «d» – сокращение от «derivative» – для обозначения производных полей. Пример производного поля – это полный путь иерархии для дашборда в «Коллекции», записанный в одну строку (поле sd_path_fine_bi в таблице z11_ui_dir4rep_hierarhy). Данный подход является скорее исключением, чем правилом.

Большая часть функций имеет внутри себя процедуру zfr_proc_log (по неймингу функция, по факту процедура), которая логирует начало и окончание работы функции, в том числе и возникающие ошибки при выполнении кода. Данная информация необходима для получения информации о процессе сборки таблиц ОМ. В дальнейшем информация используется при построении из лога отчёта о работе объектной модели.

Функция пишет данные о процессе выполнения других функций ОМ в таблицу z01_proc_log. Она использует исключительно системное время clock_timestamp, так как при использовании серверного времени всем операциям выполняемых в функциях, вызываемых из мастер-функции и других агрегатных функциях, было бы выставлено неверное значение, одинаковое для всех операций и равное времени завершения последней функции.

На скорость выполнения функций напрямую влияет размер сред, и в первую очередь количество объектов BI-системы (дашборды, компоненты, датасеты и т.д.).

Кроме того есть ещё особенности выполнения быстрых и медленных операции обработки.

Наиболее медленные операции - это извлечение данных из json встроенным функционалом postgres, а также операции с предварительным декодированием json файлов из кодировки base64 (например, получение конфигурации виджетов дашбордов для таблицы z01_wid_conf).

Потенциально длительными операциями являются операции перебора каждого значения при рекурсивной обработке списков. Например, получении информации о происхождении датасетов.

Среди таблиц ОМ существует ряд наиболее часто используемые таблицы при построении отчётности:
z11_ui_dir4rep_hierarhy - публикации дашбордов в "Коллекции"
z01_wbk_all - рабочие книги в "Песочнице"
z01_dsb_all - "Дашборды"
z01_ds_all - "Датасеты"
z01_con_list - "Подключения"
z11_usr_status - состояние УЗ пользователя
z01_rl_dep_pos_list - структура "Подразделение с позициями"
z01_clc_dir_list - каталоги "Коллекции"
z01_rl_user2dep_pos - отнесение УЗ пользователя на "Подразделение с позициями"
z01_rl_user2roles - отнесение УЗ пользователя на "Роль"
z01_upd_task_list - лог обновления датасета из "Источники"
z01_log_fre_fr_stat - лог действий бизнес-пользователей и разработчиков
z01_log_fre_view_stat - лог просмотров бизнес-пользователей и разработчиков

Таблицы z01_log_fre_fr_stat и z01_log_fre_view_stat используют данные таблицы fine_record_execute из LogDB, которая должна быть доступна в БД FineDB (например, через плагин postgres_fdw).

В ОМ имеется ряд таблиц с извлеченными данными, которые не используются активно, но потенциально могут быть использованы для построения связанной отчётности. Например, информация о настройке виджетов дашбордов в таблице z01_wid_conf. Данная операция является длительной и может быть пропущена в процессе ежедневной сборки ОМ. Сохраняется возможность запуска данной операции отдельно.

В рамках формирования таблиц ОМ была извлечена большая часть данных, но при необходимости можно дополнительно получить оставшуюся информацию об объектах, в том числе из данных формата json. При этом используются схожие с уже имеющимися в решении подходы получения данных.

Типовые действия администратора при работе с таблицами ОМ при доработке функций:
1) Изменение таблицы с использованием DDL-операции над таблицей (добавление/удаление столбца, смена имени и типа, наложение других ограничений ввода)
2) Пересоздание таблицы с использованием вставки кода SQL с дефолтными столбцами через удаление предыдущей таблицы

При пересоздании таблицы, если используемая УЗ БД для запуска функций ОМ не имеет достаточных прав, то может потребоваться перевыдача прав на использование таблиц и функций.

Часть таблиц является промежуточными и не используются напрямую в качестве источников данных для системных дашбордов. На этапе валидации расчётов было принято данное решение для лучшего контроля выполнения этапов расчётов.

Часть таблиц из числа наиболее редко используемых содержит информацию, которая может быть использована при более детальном исследовании ОМ. Например, состав и тип полей датасетов, меры рабочих книг, сетка расписаний обновлений могут быть добавлены отдельными полями в уже имеющиеся таблицы ОМ.

1.4	ЗАПУСК СБОРКИ ОМ

Сборка ОМ начинается с запуска функции мастер функции zfr_om_opt. Данная функция выполняет последовательные вызовы других функций, в основном обновляющих схожие по имени таблицы. Например, функция zfr_dsb_all обновляет данные таблицы z01_dsb_all.

Текущий список вызываемых функций со статусом запуска и порядком запуска содержится в таблице z01_om_plan.

Пример нашей реализации ниже

|group_num|sub_group_num|sub_group_name|func_name|is_enable|group_name|
|---------|-------------|--------------|---------|---------|----------|
|1|1|WBK|zfr_wbk_items|1|OM|
|1|2|WBK|zfr_wbk_hrh_table|1|OM|
--/ разрыв /--
|1|9|DSB|zfr_dsb_all|1|OM|
--/ разрыв /--
|1|12|WID|zfr_wid_all|1|OM|
--/ разрыв /--
|1|13|LNK|zfr_lnk_dsb2wid|1|OM|
|1|14|LNK|zfr_lnk_wid2ds|1|OM|
--/ разрыв /--
|1|35|DS|zfr_ds_all|1|OM|
--/ разрыв /--
|1|51|WID|zfr_wid_all|1|OM|

Поле group_num и sub_group_num используются для сортировки записей для вызова
Поле is_enable включает или исключает в общий расчёт функцию.

У функции сборки ОМ zfr_om_opt есть возможность варианты вызова: 
1) всех включенных функций при параметре mode_num = 1 и is_enable = 1
2) ручной вызов отдельных функций в виде любого where выражения для SQL параметра in_where_cond при mode_num = 0 (например, вызов функции zfr_om_opt(0,'sub_group_name = ''CON''');)

Ход выполнения расчётов каждой функции можно получить из таблицы лога z01_proc_log. Как правило используются две записи по функции: о её старте is_end = 0 и об её успешном завершении is_end = 1. При получении ошибки вместо успешного выполнения (is_end = 1) будет добавлена запись is_end = 0 и записан текст ошибки в поле error_msg.

Мастер-функция zfr_om_opt не возвращает ошибку в случаем появления ошибки в вызываемой внутри себя функции. В конце цикла обработки сверяется количество успешно завершенных этапов расчёта и их общее количество. Если они совпадают, то возвращается статус "SUCCESS", если нет - "FAILED". В случае ошибки в самой мастер-функции возвращается её код ошибки и вызывается событие ошибки.

Например, возвращаемая строка "SUCCESS mode_num: 0, in_where_cond: [sub_group_name = 'CON'] state CORR/ALL: 2/2" говорит, что все 2 вызываемые функции завершились успешно.

Если не выполнено подключение системы версионирования, то часть расчётов с использованием таблиц префикса "zst_" необходимо исключить из общего плана в таблице z01_om_plan.

Это такие функции как, например, zfr_clc_rep_hst или zfr_con_hist.

1.5	ОСОБЕННОСТИ РАЗРАБОТКИ ОМ

1) Задача по формированию расписаний на обновления датасетов типа Extract решена частично. Префикс данных таблиц zXX_upd_sch.
2) Задача по чтению лога рассылок решена в общей части через использование таблицы хранения версионированных изменений FineDB. Часть полей не вынесена в сформированную таблицу.
3) Не решена задача связи объектов в "Корзине" с реальными объектами из-за отличий id
4) При связи FineDB и LogDB с таблицей fine_record_execute используется логин, а не id пользователя
